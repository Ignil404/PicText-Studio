## Context

Проект представляет собой конструктор изображений с текстом. После реализации аутентификации (JWT, роли пользователей) необходима административная панель для оперативного управления контентом и пользователями. Текущее состояние: шаблоны хранятся в БД, но нет UI для их управления; статистика требует прямого доступа к БД.

## Goals / Non-Goals

**Goals:**
- Единая админ-панель с навигацией по разделам (Dashboard, Templates, Users, Stats)
- Полный CRUD для шаблонов с inline preview
- Просмотр статистики: рендеры по времени, популярные шаблоны, активность пользователей
- Управление пользователями: поиск, фильтрация, блокировка
- Ролевая модель с разделением прав (user/admin)
- Защита всех admin endpoints проверкой JWT + роли

**Non-Goals:**
- Редактор шаблонов с drag-and-drop (только форма с полями)
- Real-time аналитика (агрегированные данные)
- Экспорт данных в CSV/PDF
- Массовые операции с пользователями (batch actions)
- Управление системными настройками (env vars)

## Decisions

**1. Layout: Sidebar + Content Area**
- Решение: Фиксированный сайдбар слева (200px), основной контент справа
- Обоснование: Стандартный паттерн для админ-панелей, легкая навигация между разделами
- Альтернативы: Топ-навигация (менее масштабируема), Drawer на мобильных (не приоритет)

**2. Admin Check: Dependency + Middleware**
- Решение: `require_admin()` dependency для FastAPI + `AdminGuard` компонент на фронтенде
- Обоснование: Единая точка проверки на бэкенде, чистая защита роутов на фронтенде
- Альтернативы: Проверка в каждом endpoint (дублирование), только frontend check (небезопасно)

**3. Stats Aggregation: SQL + Caching**
- Решение: Агрегация на уровне SQL (COUNT, GROUP BY), кеширование в Redis для тяжёлых запросов
- Обоснование: Простота реализации, достаточно для текущих объёмов
- Альтернативы: ClickHouse/OLAP (переусложнение), materialized views (не нужно сейчас)

**4. Template Form: Multi-step vs Single-page**
- Решение: Single-page форма с секциями (Basic, Style, Preview)
- Обоснование: Быстрый доступ ко всем полям, меньше кликов
- Альтернативы: Wizard (для новичков, но медленнее)

**5. User List: Server-side Pagination + Search**
- Решение: Пагинация на бэкенде (limit/offset), поиск по email через ILIKE
- Обоснование: Масштабируемость при росте пользователей
- Альтернативы: Client-side pagination (не работает для больших списков)

## Risks / Trade-offs

**[Risk] Админ удалит активный шаблон → ломаются ссылки на превью**
→ Mitigation: Soft delete (is_active флаг) вместо hard delete, или проверка использования

**[Risk] Тяжёлые запросы статистики блокируют БД**
→ Mitigation: Кеширование в Redis (TTL 5 мин), индексы на created_at, async queries

**[Risk] Роль admin в JWT — клиент может подделать**
→ Mitigation: Верификация на бэкенде при каждом запросе, role в JWT подписана секретом

**[Risk] Много новых endpoint'ов — растёт поверхность атаки**
→ Mitigation: Единый prefix `/api/admin/*`, middleware проверки роли на уровне router

**[Trade-off] Поля шаблона в JSON vs отдельные колонки**
→ Выбрано: Структурированные поля (name, gradient_start, etc.) + config JSON для extras
→ Причина: Основные поля нужны для фильтрации/сортировки

## Migration Plan

1. **Подготовка**: Создать первого admin пользователя (ручная миграция или seed)
2. **Database**: Миграция добавления `role` column (default 'user')
3. **Backend**: Деплой новых admin routers (не ломает существующий функционал)
4. **Frontend**: Деплой admin страниц (не видны обычным пользователям)
5. **Smoke test**: Проверка доступа admin, недоступности для user

Rollback: Удаление admin routers, откат миграции (downgrade Alembic)
